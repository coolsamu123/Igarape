# Migration Plan: Transitioning Igarape Project to Subapp Data Source

## Objective
The Igarape Project currently relies on raw descriptive fields ("Description" and "Remarks") sourced from an uploaded Excel file to power its UI views and Gemini-based intersection analysis. 

The goal of this migration is to **shift the source of truth** for project characteristics to the deeply extracted, structured data generated by the new **CIOO Project Goals Extractor Sub-App**. 

Instead of raw descriptions, the main application will now use **8 specific structured dimensions** (Digital Technologies, Change Management, Security Impacts, Regional Impacts, AI Embedded, GIO SL/DDS Impacts, DDS/GIO Workload, Business Apps & CIs) to render project details and power the Gemini API matching/intersection engine.

---

## Architecture & Data Flow Changes

### Current State
1. Upload Excel -> `projects` table (SQLite)
2. UI fetches `ProjectSummary` (Deduplicated rows + `description` + `remarks`)
3. Gemini API compares Project A vs Project B using raw descriptions and remarks.

### Future State
1. Sub-app processes Drive documents -> `project_goals` table (SQLite) with 8 AI-extracted fields.
2. Excel is still uploaded for metadata (Costs, Dates, Decisions), but joined with `project_goals`.
3. UI fetches `ProjectSummary` enriched with the 8 fields from `project_goals` via a `LEFT JOIN`.
4. Gemini API compares Project A vs Project B by explicitly analyzing overlaps/conflicts across the 8 structured dimensions.

## Phase 1: Data Model & API Integration

### 1. Update TypeScript Definitions (`src/lib/types.ts`)
Extend the `ProjectSummary` interface to include the newly available fields from the subapp.
```typescript
export interface ProjectSummary {
  // ... existing fields (projectId, name, dds, costKEur, etc.)
  
  // New Sub-App fields
  digitalTechnologies?: string;
  changeManagement?: string;
  securityImpacts?: string;
  regionalImpacts?: string;
  iaEmbedded?: string;
  gioSlDdsImpacts?: string;
  ddsGioWorkload?: string;
  businessAppsCis?: string;
  
  subappAnalyzed: boolean; // Flag to indicate if subapp data exists for this project
}
```

### 2. Update the Projects API Route (`src/app/api/projects/route.ts`)
Modify the main `GET` request to perform a join between the `projects` table and the `project_goals` table.
- Read from `projects` as the base to keep Excel metadata (like costs and dates).
- `LEFT JOIN project_goals pg ON p.project_id = pg.project_id`
- Map the resulting rows to the updated `ProjectSummary` model, populating the 8 subapp fields.

### 3. Update Similarity Engine (`src/lib/similarity.ts`)
Modify the `extractTags()` fallback logic. Instead of finding tags only in `description`, it should extract key terms from `digitalTechnologies`, `businessAppsCis`, and `securityImpacts` to generate better initial Jaccard similarities before the AI runs.

## Phase 2: Upgrading the Gemini API Matching Engine

This is the core of the request. The Gemini intersection analysis must be updated to consume the structured data rather than free-form descriptions.

### 1. Modify the Analysis Endpoint (`src/app/api/analyze/route.ts`)
Update the prompt generation logic for Pairwise and Cluster analysis.

**Old Prompt Context:**
> Project A: {name}
> Description: {description}
> Remarks: {remarks}

**New Prompt Context:**
> Project A: {name} ({dds})
> - Digital Technologies: {digitalTechnologies}
> - AI/IA Embedded: {iaEmbedded}
> - Security Impacts: {securityImpacts}
> - Change Management: {changeManagement}
> - Regional Impacts: {regionalImpacts}
> - GIO SL/DDS Impacts: {gioSlDdsImpacts}
> - DDS/GIO Workload: {ddsGioWorkload}
> - Business Apps & CIs: {businessAppsCis}

### 2. Adjust System Prompt Instructions
Instruct Gemini to specifically look for matching constraints or overlaps in:
- **Technological Synergies**: Matching elements in "Digital Technologies" and "Business Apps & CIs".
- **Operational Bottlenecks**: Overlapping constraints in "DDS/GIO Workload" or "GIO SL/DDS Impacts".
- **Risk Multipliers**: Compounding complexities in "Security Impacts" or "Change Management".

## Phase 3: Updating the User Interface (UI)

### 1. Detail View Component (`src/components/DetailView.tsx`)
Redesign the side panel or modal that opens when a project is clicked.
- Remove the large generic "Description" / "Remarks" text blocks.
- Implement a 2-column grid or an Accordion view displaying the 8 structured fields clearly.
- Add visual indicators (e.g., a chip/badge) if `iaEmbedded` is active ("AI Powered"), or if `securityImpacts` is high.

### 2. Graph & Matrix Views (`src/components/GraphView.tsx`, `MatrixView.tsx`)
- Update the tooltip on hover to surface critical subapp data (e.g., "AI: Yes", "Impacts: GIO Network").
- *Optional Feature*: Add coloring or grouping modes to the Graph View (e.g., "Group by AI Embedded" or "Color by Security Impact").

### 3. Sidebar KPIs & Filters (`src/components/Sidebar.tsx`)
Add new filtering capabilities based on the subapp data:
- Checkbox: "Show only AI-embedded projects"
- Filter: "Filter by Business App CI (Search)"
- KPI counter: "X Projects with GIO/DDS Workload Impacts"
